# SMAP 平台新增页面缺口清单（v2）

> **分析日期**：2026-05-14
> **分析视角**：业务操作系统，AI Native 范式
> **核心前提**：数据/集成/管理基础设施已有；企业有自己的作业票/审批系统，平台只对接数据

---

## 一、范式修正：从"流程驱动"到"智能体驱动"

### 上一版分析的问题

| 上一版思路 | 修正后思路 | 修正原因 |
|-----------|----------|---------|
| 建一个独立的"流程管理"模块 | 流程编排 = 数字员工编排，合并在数字员工模块内 | 不存在脱离Agent的"纯流程"，流程的本质是Agent在干活 |
| 需要BPM流程编排画布（像Dify） | 用自然语言创建和验证（像OpenClaw的Skill机制） | 自然语言编排才是AI Native，画布只是传统思维 |
| 需要独立的"审批中心"模块 | 审批是企业的业务，平台只对接审批结果数据 | 企业有自己的审批系统，平台不应该重复建设 |
| 审批详情是独立页面 | 审批结果体现在任务详情中 | 任务详情页已有执行记录和状态，审批只是任务的一种状态 |
| 每个Agent需要独立的Prompt管理页面 | Skill是数字员工的能力载体，Skill管理 = 数字员工配置 | 不按"传统Agent"思路拆页面，按"数字人+Skill"思路组织 |

### 核心模型：Skill 驱动的虚拟数字员工

```
数字员工（虚拟人）= 角色定义 + Skill 集合 + 记忆 + 工具权限

Skill 的本质：
  - 一个 Skill = 一个自然语言指令集（Markdown）
  - 触发方式：关键词匹配 / 定时触发 / 事件触发 / 手动调用
  - 执行方式：Agent 按照 Skill 指令自动执行
  - 组合方式：多个 Skill 可以组成一个业务流程

示例：小安（报警诊断专员）
  ├ Skill-1：报警分级判定（触发：GDS报警事件）
  ├ Skill-2：根因分析（触发：报警分级为紧急时）
  ├ Skill-3：处置建议生成（触发：根因分析完成后）
  └ Skill-4：通知推送（触发：处置建议生成后）
  → 这4个Skill串起来，就是一条"报警处置流程"
  → 不需要额外的"流程编排画布"
```

### 维度一：数字员工能力（共5页）

> 核心理念：流程编排 = Skill 组合 = 数字员工配置
> 不需要独立的"流程管理"模块，全部整合在数字员工模块内

#### 1.1 Skill 管理中心
- **路径**：`pages/ai-staff/skills.html`
- **所属模块**：数字员工（L2新入口）
- **功能描述**：管理数字员工的全部 Skill 列表。展示每个 Skill 的状态（启用/禁用）、所属数字员工、触发方式、最近执行情况。支持按数字员工筛选、按触发类型筛选（定时/事件/关键词/手动）。
- **与现有页面的关系**：与 roster.html（花名册）并列；点击 Skill 可进入编辑页；与 profile.html 的"能力标签"区联动
- **优先级**：**P0**
- **为什么**：Skill 是数字员工干活的核心载体，没有 Skill 管理页面，数字员工就是空壳

#### 1.2 Skill 创建/编辑页
- **路径**：`pages/ai-staff/skill-edit.html`
- **所属模块**：数字员工 → Skill管理中心 → 子页面
- **功能描述**：**用自然语言创建和编辑 Skill**。核心功能：
  - 左侧：Skill 结构编辑（描述 / 使用方法 / 指令）— 纯 Markdown 编辑
  - 右侧：预览与验证面板 — 输入模拟场景，看 Skill 是否被正确触发、执行结果是否符合预期
  - 关联配置：绑定数字员工、设置触发条件（关键词/定时Cron/事件源）、配置工具权限（可调用哪些API/查询哪些数据）
  - 版本管理：Skill 的历史版本对比和回滚
- **与现有页面的关系**：skills.html 的编辑页；创建完成后，Skill 自动挂载到对应的数字员工上
- **优先级**：**P0**
- **为什么**：这是整个平台最核心的页面——用自然语言定义业务能力，验证后即可上线。是 OpenClaw Skill 机制的可视化落地

#### 1.3 Skill 协作编排
- **路径**：`pages/ai-staff/skill-pipeline.html`
- **所属模块**：数字员工 → Skill管理中心 → 子页面
- **功能描述**：编排多个 Skill 的串行/并行/条件协作链路。
  - 方式一（推荐）：**自然语言描述** → 系统自动解析为 Skill DAG。例如输入"当收到GDS紧急报警时，先执行报警分级判定，然后根据级别执行不同的处置流程"，系统自动生成编排图
  - 方式二：可视化 DAG 编辑器（作为辅助方式）
  - 支持跨数字员工编排（小安的 Skill → 小研的 Skill → 小报的 Skill）
  - 保存为"业务场景模板"，可一键启用
- **与现有页面的关系**：skills.html 的编排视图；编排完成后生成新的 Skill Pipeline，可作为独立单元挂载到花名册
- **优先级**：**P0**
- **为什么**：多 Skill 协作 = 多数字员工协作 = 业务流程。这是平台从"单点AI"到"AI操作系统"的关键跳板

#### 1.4 数字员工记忆管理
- **路径**：`pages/ai-staff/memory.html`
- **所属模块**：数字员工（L2新入口）
- **功能描述**：管理数字员工的记忆系统：
  - **经验记忆**：历史诊断结论和处理方案（结构化存储，可检索、可复用）
  - **知识记忆**：关联的知识库文档片段（向量索引，支撑RAG检索增强）
  - **会话记忆**：当前活跃对话的上下文窗口管理
  - 记忆质量评估：记忆的准确率、被引用次数、过期清理
- **与现有页面的关系**：与 profile.html 并列；影响数字员工的诊断准确度和个性化
- **优先级**：P1
- **为什么**：没有记忆的数字员工，每次处理同类问题都从零开始。记忆积累是数字员工"越用越聪明"的关键

#### 1.5 数字员工工作台
- **路径**：`pages/ai-staff/dashboard.html`
- **所属模块**：数字员工（L2新入口）
- **功能描述**：数字员工的"运营视角"面板：
  - 全局概览：所有数字员工的实时状态（工作中/空闲/异常）
  - 执行统计：每个数字员工的今日执行次数、成功率、平均耗时
  - Skill 运行记录：最近执行的所有 Skill 及其结果
  - 消息推送记录：数字员工主动推送了哪些消息给哪些人
  - 异常监控：执行失败、超时、低置信度输出的告警
- **与现有页面的关系**：与 roster.html（花名册，侧重"介绍"）互补，本页侧重"运营监控"
- **优先级**：P1
- **为什么**：管理员需要知道数字员工整体运行情况，而不能一个个点进去看

### 维度二：交互能力（共3页）

> 核心理念：消息不只是系统通知，而是**数字员工在工作过程中主动推送的结果**
> 需要讨论的核心问题：什么情况下、什么样的数字员工可以推送什么样的消息？

#### 2.1 消息中心
- **路径**：`pages/message/center.html`
- **所属模块**：工作平台（L2新入口）
- **功能描述**：统一消息枢纽，按来源分类：
  - **数字员工消息**：小安的诊断结果推送、小报的报告完成通知、小巡的巡检异常提醒等
  - **任务消息**：任务分配、到期提醒、状态变更
  - **系统消息**：平台公告、版本更新、维护通知
  - 每条消息标注：来自哪个数字员工 / 关联哪个任务 / 可点击跳转到对应详情页
- **与现有页面的关系**：替代 index.html 的滚动 ticker；工作台的对话视图可调用消息推送
- **优先级**：**P0**
- **为什么**：当前报警靠滚动ticker，紧急情况不可靠。更重要的是，数字员工处理完事情后需要一个通道把结果"交"给人

#### 2.2 消息权限配置
- **路径**：`pages/message/rules.html`
- **所属模块**：消息中心 → 子页面
- **功能描述**：配置**数字员工的消息推送权限和规则**：
  - 哪些数字员工可以推送消息？（小安可以，但小库的普通检索结果不需要推送）
  - 什么级别的内容需要推送？（紧急报警 → 立即推送 + 强提醒；日常巡检报告 → 仅站内信）
  - 推送给谁？（按角色/按人员/按值班表配置推送对象）
  - 推送渠道？（站内信 / 浏览器弹窗 / 企微 / 短信 / 电话）
  - 免打扰规则？（夜间仅保留紧急报警推送）
  - **关键设计**：每条推送消息附带"置信度标签"（高/中/低），低置信度的AI结论标记为"建议参考"而非"处置指令"
- **与现有页面的关系**：message/center.html 的设置页；与数字员工的 Skill 执行结果联动
- **优先级**：**P0**
- **为什么**：这是你提到的核心问题——数字员工能推送什么消息、什么情况下推送，需要可配置。否则要么信息轰炸，要么关键信息遗漏

#### 2.3 全局搜索
- **路径**：`pages/search/global.html`
- **所属模块**：工作平台（L2新入口，建议做成全局浮层 Cmd+K）
- **功能描述**：跨模块统一搜索入口，索引范围：
  - 任务（按状态/类型/时间搜索）
  - 数字员工对话记录（按对话内容搜索）
  - 报警记录（按装置/级别/时间搜索）
  - 文档/知识库（全文检索）
  - 数字员工产出物（报告/诊断结论/巡检记录）
  - Skill 列表（按名称/功能描述搜索）
- **与现有页面的关系**：与 workspace.html 并列；可通过 index.html 的顶部搜索框或快捷键唤起
- **优先级**：P1
- **为什么**：用户想找"上个月液氨储罐的所有报警记录"，不知道去哪个模块。全局搜索是操作系统级标配

## 七、需要讨论的开放问题

### 问题1：数字员工的消息推送权限模型

这是你提到的核心问题。建议的讨论框架：

```
推送决策矩阵：

数字员工类型     可以推送的内容            推送条件              推送级别
─────────────────────────────────────────────────────────────────
小安(报警诊断)  报警分级结论、处置建议    报警事件触发          紧急/高
小研(风险评估)  风险评估报告            评估完成后            普通
小报(报告生成)  月度/周报完成通知        报告生成完成后        普通
小巡(智能巡检)  巡检异常发现            巡检中发现异常时      高
小库(知识检索)  ⚠️ 不主动推送           仅在被其他Skill调用时  -
小识(智能问答)  ⚠️ 不主动推送           仅在对话中回复       -
小审(作业审批)  审批建议结果            被调用时              中
```

**关键设计原则**：
- 不是所有数字员工都能主动推送消息（小库、小识这类工具型的不推送）
- 推送消息必须附带**来源标识**（哪个数字员工）和**置信度标签**（高/中/低）
- 低置信度推送标记为"建议参考"，高置信度才标记为"处置指令"
- 值班场景下只有紧急级别消息才突破免打扰

### 问题2：Skill 的创建方式

两种路径需要讨论：

| 路径A：自然语言创建 | 路径B：模板+微调 |
|-------------------|----------------|
| 用户直接用自然语言描述 Skill | 平台提供常见 Skill 模板（报警处置/巡检分析/报告生成） |
| AI 自动解析为结构化 Skill | 用户基于模板修改细节 |
| 门槛低但准确性需验证 | 准确性高但灵活性低 |
| 适合技术用户 | 适合业务用户 |

**建议**：两种都支持。页面默认展示模板库（路径B），同时提供"用自然语言创建"入口（路径A）。
